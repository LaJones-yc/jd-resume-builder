# Turning source material into resume evidence

## Existing resume or CV

Extract all text, identify sections and entries, and create draft library records. Treat resume claims as user-provided evidence but ask about vague ownership, missing dates, and unsupported metrics. Preserve both language versions when available.

## Paper or thesis

Extract the research question, dataset, methods, baselines, evaluation metrics, results, artifacts, collaborators, and publication/award status. Then ask what the user personally designed, implemented, analyzed, wrote, or presented. Convert the confirmed contribution into 2-4 bullets rather than summarizing the whole paper.

## Slide deck or project report

Separate problem framing, proposed plan, actual execution, deliverables, and measured outcomes. Slides often mix these states. Ask which items were completed and which belonged to the user.

## Code repository

Inspect README, architecture, dependency files, core modules, tests, issues, commits, releases, and deployment configuration. Derive demonstrable technical work such as implemented components, stack, data flow, scale visible in fixtures/logs, and test coverage. Do not infer user growth, revenue, adoption, performance improvement, team leadership, or individual ownership without evidence.

## From source to bullets

1. List atomic facts with citations to files/pages/commits.
2. Separate project-level facts from personal contributions.
3. Ask targeted questions about ownership, decisions, constraints, scale, and outcomes.
4. Propose 2-4 candidate bullet angles: technical depth, product judgment, research rigor, leadership, or measurable impact.
5. Let the user choose or correct the decomposition when multiple narratives are reasonable.
6. Save only confirmed claims as final bullets.

## Useful questions

- Which part did you personally own or lead?
- What was the starting problem or limitation?
- What alternatives did you evaluate, and why did you choose this method?
- What artifact did you deliver?
- Was it deployed, tested, published, presented, or only prototyped?
- What exact metric changed, over what baseline and evaluation set?
- How large was the dataset, user group, team, or operating scale?
